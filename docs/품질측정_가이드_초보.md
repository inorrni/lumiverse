# 품질 측정 입문 가이드

> [!abstract] 이 문서는
> [[품질점검_20260618]]의 S1~S3(브라우저 실측)을 **처음 해보는 사람이 클릭 단위로 따라 하도록** 풀어 쓴 안내서다. 핵심 원칙은 하나: **추측하지 말고 측정한다.** 보기 좋게 고치기 전에, 숫자로 Before를 찍고 → 고치고 → After를 다시 찍는다.

## 0. 공통 준비 (5분)

1. **Chrome** 설치(가장 도구가 잘 갖춰져 있음).
2. **개발자도구(DevTools) 여는 법**:
   - Mac: `Cmd + Option + I` / Windows: `F12`
   - 또는 화면 우클릭 → **검사(Inspect)**.
   - 도구창 상단 탭: `Elements / Console / Network / Performance / Lighthouse …`
3. **React Developer Tools 확장** 설치(S2에서 필요):
   - Chrome 웹스토어에서 "React Developer Tools" 검색 → 설치.
   - 설치 후 DevTools에 **`Components`·`Profiler`** 탭이 생긴다.
4. **Node/npx**(S3 CLI에서 필요): 터미널에서 `node -v` 가 나오면 OK(이미 설치돼 있음).

> [!tip] 어디서 측정하나 — 배포 vs 로컬
> - **점수(Lighthouse)·실사용 체감**은 **배포 사이트**(https://inorrni.github.io/lumiverse/)에서.
> - **리렌더(Profiler)**는 **로컬 dev**(`npm run dev` → http://localhost:5173)에서가 편하다(소스가 매핑돼 컴포넌트 이름이 보임).
> - 로그인이 필요한 화면(/app/*)은 측정 전에 먼저 로그인해 둔다.

## 1. S1 — Network (느린 호출 찾기)

**목표**: 1초 이상 걸리는 호출 1개를 찾는다.

### 따라 하기
1. DevTools → **`Network`** 탭.
2. 상단 필터에서 **`Fetch/XHR`** 클릭(이미지·폰트 빼고 API만 본다).
3. 왼쪽 위 **🚫(Clear)** 로 목록 비우기. **`Preserve log`** 체크(페이지 이동해도 기록 유지).
4. 앱에서 **실제로 동작**시킨다:
   - 목표 만들기(`goal-breakdown` 호출) / 점검 탭 들어가기(`mid-check`·`self-insight`) / 로그인(Supabase auth).
5. 표의 **`Time`** 컬럼 헤더를 눌러 **내림차순 정렬** → 맨 위가 가장 느린 호출.
6. **1초(1000ms) 넘는 항목**을 클릭 → 오른쪽 패널에서 원인 분류:
   - **Timing** 탭: `Waiting for server response (TTFB)`가 길면 → **서버(Edge Function/LLM)가 느림**.
   - **Size**가 크면 → 응답이 큼(페이로드 줄이기).
   - 여러 호출이 **계단식(워터폴)**으로 줄줄이면 → 직렬 호출, `Promise.all`로 병렬화 여지.

### Slow 3G로 로딩 UI 확인
1. Network 탭 상단 **`No throttling`** 드롭다운 → **`Slow 3G`** 선택.
2. 주소창에 `/app/universe` 진입(또는 새로고침).
3. **three.js 청크(~242KB gzip)** 받는 동안 **"우주를 그리는 중…"** 폴백이 보이는지 확인 → 보이면 정상(로딩 UX OK).
4. 확인 끝나면 throttling을 **`No throttling`** 으로 되돌린다.

> [!tip] 프론트 문제인지 서버 문제인지 가르기
> 느린 호출 우클릭 → **`Copy as cURL`** → 터미널에 붙여 실행. 터미널에서도 느리면 **서버(백엔드)** 문제, 빠르면 **프론트** 문제.

### 무엇이 "좋음"인가
- 일반 API: < 300ms 면 양호, 1s+ 면 점검 대상.
- LLM 호출(goal-breakdown/mid-check)은 원래 수 초 걸릴 수 있음 → **로딩 UI가 잘 떠 있으면** 통과로 본다(체감 관리).

## 2. S2 — React Profiler (불필요한 리렌더 찾기)

**목표**: 클릭/조작 시 **불필요하게 다시 그려지는 컴포넌트 1개**를 찾는다.

### 2-1. 눈으로 보기 (Highlight)
1. DevTools → **`Components`** 탭 → 우측 **⚙️(톱니)**.
2. **`Highlight updates when components render`** 체크.
3. 앱을 조작(투두 체크, 입력, 우주 회전) → **다시 그려지는 영역이 색 테두리로 깜빡**인다.
4. 조작과 **상관없는 영역**이 깜빡이면 = 불필요한 리렌더 후보.

### 2-2. 녹화해서 보기 (Profiler)
1. DevTools → **`Profiler`** 탭 → 왼쪽 **● (녹화)** 클릭.
2. 측정할 동작 수행(예: `/app/universe`에서 **회전·줌 몇 초**).
3. **■ (정지)**.
4. 상단 보기를 **`Ranked`** 로 → commit(렌더 1회)에서 **오래 걸린 컴포넌트가 위**로.
5. 컴포넌트 클릭 → 오른쪽 **"Why did this render?"** 로 원인 확인.

### 원인별 처방 (※ 측정으로 진짜 느릴 때만!)
| 원인 표시 | 처방 |
|---|---|
| Parent re-rendered | 자식을 `React.memo`로 감싼다 |
| Props changed (매번 새 함수/객체) | `useCallback` / `useMemo`로 안정화 |
| Context changed | context value를 `useMemo`로 안정화 |
| Hooks order changed | **버그** — Hook을 조건문 밖 최상단으로 |

> [!warning] 조기 최적화 금지
> `memo/useMemo/useCallback`도 비용이 있다. **Profiler로 느린 곳을 확인한 뒤에만** 적용한다. 이 프로젝트는 이미 스토어 context를 `useMemo`로 안정화해 둬서, 큰 문제는 우주 탭 3D(`UniverseScene`/`Galaxy3D`) 회전 시 정도만 보면 된다.

### Performance 탭으로 끊김(Long Task) 보기
1. DevTools → **`Performance`** → ● 녹화 → 우주 회전 → ■ 정지.
2. 타임라인의 **빨간 삼각형/막대(Long Task, 50ms+)** 가 많으면 끊김.
3. `Bottom-Up` 보기로 가장 시간 먹는 함수 확인.

## 3. S3 — Lighthouse (4지표 점수)

**목표**: Performance / Accessibility / Best Practices / SEO **4지표 점수**를 기록하고, 가장 낮은 걸 올린다.

### 방법 A — DevTools에서 (쉬움)
1. **배포 사이트**(https://inorrni.github.io/lumiverse/)를 연다.
2. DevTools → **`Lighthouse`** 탭.
3. **Mode**: `Navigation` / **Device**: 먼저 **`Mobile`** / Categories 4개 체크.
4. **`Analyze page load`** 클릭 → 1~2분 후 점수 리포트.
5. **Device를 `Desktop`으로** 바꿔 한 번 더.

### 방법 B — 터미널에서 (CLI)
```bash
npx lighthouse https://inorrni.github.io/lumiverse/ --view --preset=desktop
# 모바일도:
npx lighthouse https://inorrni.github.io/lumiverse/ --view
```
- `--view` 는 끝나면 HTML 리포트를 자동으로 열어 준다.

### 4지표 목표 & 읽는 법
| 지표 | 목표 | 자주 나오는 개선점 |
|---|---|---|
| Performance | 80+ | 이미지 최적화(이미 완료), 코드 스플리팅(이미 lazy), 폰트 `display=swap` |
| Accessibility | 95+ | `<img>` alt(완료), 라벨 연결, 색대비 4.5:1, 포커스 표시 |
| Best Practices | 95+ | HTTPS(OK), 콘솔 에러 없음 |
| SEO | 95+ | title·description(OK)·OG(OK)·canonical(OK) |

- **Core Web Vitals**(Performance 안): **LCP < 2.5s · INP < 200ms · CLS < 0.1**. 빨간색이면 개선 1순위.
- 리포트의 **"Opportunities / Diagnostics"** 에 무엇을 고치면 몇 초/몇 점 오르는지 친절히 나온다.

### 개선 루프
**측정 → 가장 낮은 지표의 상위 3개 개선점 적용 → 재측정 → 5점 이상 올랐는지 확인.** 추측으로 손대지 말 것.

## 5. 자주 막히는 점 (FAQ)

- **Network에 아무것도 안 떠요** → 측정 전에 목록을 비웠는지, `Fetch/XHR` 필터 때문에 가린 건 아닌지 확인. 새로고침하면 처음부터 다시 잡힌다.
- **Components/Profiler 탭이 없어요** → React Developer Tools 확장 미설치. 설치 후 DevTools를 닫았다 다시 연다.
- **배포에서 Profiler 컴포넌트 이름이 이상해요** → 프로덕션 빌드는 이름이 축약됨. 리렌더는 **로컬 dev**에서 보는 게 정확.
- **Lighthouse 점수가 잴 때마다 달라요** → 정상(네트워크·머신 상태 영향). 2~3번 재서 **중앙값**을 쓴다. 확장 프로그램 영향 피하려면 **시크릿 창**에서 측정.
- **로컬은 되는데 배포가 달라요** → 배포는 `/lumiverse/` 하위경로. 점수는 꼭 **배포 URL**로 측정.

> [!note] 이 프로젝트 현재 상태
> 이미지·코드스플릿·테스트·alt·OG·canonical은 코드로 닫힌 상태(품질점검 문서 참고). 이 가이드로는 **숫자 확인 + 우주 탭 3D 체감**만 떠서 Before/After를 채우면 된다.
