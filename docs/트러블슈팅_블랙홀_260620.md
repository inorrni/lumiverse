# 트러블슈팅 — 블랙홀 기능 (2026.06.20)

> [!note] 개요
> 블랙홀 관련 보고 3건을 정리한다. 1·3은 같은 뿌리(대시보드 Today 카드의 표시 한계), 2는 별도(은하·행성 보관 축 불일치). 핵심 교훈: **"안 보인다 ≠ 데이터가 없다"** — 새 행성은 DB·우주맵·"모두 보기"에는 멀쩡히 존재했고, 잘린 건 카드 한 곳뿐이었다.

---

## B-1 · 보완으로 추가한 행성이 Today 카드에 안 보임

> [!bug] 증상
> 중간점검 → 보완하기에서 기존 행성을 블랙홀로 보내고 추천 행성을 추가한 뒤, 메인화면(`/app`) **Today 카드**에 새로 담은 행성이 보이지 않는다.

**재현**
1. 행성이 3개 이상인 목표에서 중간점검 진입.
2. 보완하기 → 일부 행성 유지 + 추천 행성 추가 → 보완 적용.
3. `/app` 대시보드 Today 카드 확인 → 새 행성 없음.

**원인**
- 새 행성 자체는 정상: `addPlanet`([GoalStore.jsx](../src/store/GoalStore.jsx))이 `오늘~디데이` 별과 함께 생성 → 우주맵·"모두 보기"(TodayPage)에는 보임.
- 대시보드 Today 카드([DashboardPage.jsx](../src/pages/DashboardPage.jsx))가 `activeGoal.steps.slice(0, 3)`로 **앞 3개만** 렌더. 새 행성은 `order_idx` 최댓값이라 정렬상 맨 뒤 → 3개 초과 시 **잘림**.
- 부수 문제: 카드 목록은 `todayStarId` 필터 없이 slice, 헤더 카운트(`todayTotal`)는 `todayStarId` 기준 → **목록과 카운트 불일치** 가능.

**해결**
- 단일 소스 `todaySteps = activeGoal.steps.filter(s => s.todayStarId)` 도입.
- 카드 목록을 `slice(0, 3)` → `todaySteps`(전부)로 교체, 헤더 카운트도 동일 소스 사용.
- `todaySteps`가 비면 "오늘 배정된 별이 없어요." 한 줄 표시. "모두 보기 →"는 유지.

```diff
- const todayDone = activeGoal ? activeGoal.steps.filter((s) => s.checkedToday).length : 0
- const todayTotal = activeGoal ? activeGoal.steps.filter((s) => s.todayStarId).length : 0
+ const todaySteps = activeGoal ? activeGoal.steps.filter((s) => s.todayStarId) : []
+ const todayDone = todaySteps.filter((s) => s.checkedToday).length
+ const todayTotal = todaySteps.length
...
- {activeGoal.steps.slice(0, 3).map((s, i, arr) => ( ... ))}
+ {todaySteps.length === 0 ? <p className={styles.todayEmpty}>…</p> : todaySteps.map((s, i, arr) => ( ... ))}
```

**회귀 방지** — 카드 목록·카운트가 한 소스(`todaySteps`)에서 파생되어 영구적으로 일치. 보완 추가 행성은 오늘 별을 가지므로 항상 포함.

## B-2 · 포기한 목표 은하가 블랙홀에 안 보이고 복구 불가

> [!bug] 증상
> 중간점검 → 마무리하기로 목표를 포기한 뒤 블랙홀(`/app/blackhole`)에 가면 그 은하가 보이지 않는다. 되살릴 방법도 없다.

**재현**
1. 목표 중간점검 → (이탈위험 등) 마무리하기.
2. 블랙홀 화면 진입 → 포기한 은하 없음.

**원인**
- `removeGoal`([GoalStore.jsx](../src/store/GoalStore.jsx))은 **은하**의 `status='blackhole'`만 토글(소속 행성은 `active` 유지).
- `BlackHolePage`는 `loadBlackholePlanets`로 **행성만** 조회 + 쿼리가 `galaxy.status === 'active'` 행성만 통과.
- ⇒ 포기 은하는 (은하=blackhole, 행성=active) **행성 보관함 조건에도 안 맞고, 은하를 보는 경로 자체가 없음** → 완전 미노출 + 복구 불가.

**해결**
- 스토어에 은하 축 함수 추가: `loadBlackholeGalaxies()`(status='blackhole' 은하 조회), `restoreGalaxy(id)`(active 복귀 + `reload()`).
- `BlackHolePage`가 은하+행성을 `Promise.all`로 동시 로드, "보관된 목표"·"보관된 행성" 2섹션으로 렌더, 각각 선택 Set + 통합 되살리기. 빈 상태는 둘 다 비었을 때만.
- 은하 복구 시 소속 행성은 이미 `active`라 함께 재등장.

**회귀 방지** — 보관함이 보관의 두 축(은하·행성)을 모두 조회하므로 포기 은하가 빠지지 않음. 행성 보관함의 `부모 active 필터`는 유지(포기 은하 내부 행성을 중복 노출하지 않음).

## B-3 · 행성 복구 후에도 새 행성이 Today에 안 보임

> [!bug] 증상
> 블랙홀에서 행성을 복구해 우주로 돌려도, 보완 때 추가한 새 행성이 Today에 보이지 않는다.

**원인 · 해결** — **B-1과 동일**. 행성 복구(`restorePlanets`)는 정상 동작하며 복구된 행성도 재등장한다. "새 행성이 안 보인다"는 부분은 Today 카드의 `slice(0, 3)` 잘림이 원인이었고 B-1 수정으로 함께 해소된다.

---

> [!tip] 공통 교훈
> - 표시 결함은 **데이터 경로(스토어·DB)부터 의심하기 전에, 같은 데이터를 쓰는 다른 화면(우주맵·"모두 보기")과 비교**하면 빠르게 View 문제로 좁힐 수 있다.
> - 목록과 카운트는 **같은 파생 소스**에서 만들어 불일치 가능성을 구조적으로 제거.
> - "보관(블랙홀)"처럼 **여러 엔티티(은하·행성)가 같은 상태값을 공유**하면, 조회/복구 화면이 그 축을 **모두** 다루는지 점검.
