# 셋업 · 빌드 · 배포 가이드

> [!abstract] 이 문서로 할 수 있는 것
> 새 팀원이 **클론 → 실행 → (백엔드) → 빌드 → 배포**까지 막힘 없이 따라오게 하는 한 장짜리 안내.
> 스택: **Vite 5 · React 18 · React Router 7 / Supabase(PostgreSQL·Auth·Edge Functions) · Solar(LLM)**.

---

## 0. 사전 준비

| 필요 | 버전/비고 |
|---|---|
| Node.js | **20 이상** (CI는 20) · `node -v` |
| npm | Node 동봉 |
| Git | GitHub 저장소 접근 권한 |
| Supabase 계정 | 프로젝트 **소유자만** 필요. 팀원은 키 2개만 받으면 됨 |

> [!important] 백엔드는 공유 단일 프로젝트 (먼저 이해)
> 개발·배포 **모두 소유자(inorrni)의 Supabase 프로젝트 하나**를 공유해서 쓴다.
> 팀원은 본인 프로젝트를 만들 필요 없이 **소유자가 준 키 2개**만 `.env.local` 에 넣으면 된다(§2).
> - 백엔드 셋업(스키마·LLM 키·함수·인증)은 **소유자가 1회만** 한다(§4). 팀원은 건너뛴다.
> - 로그인 계정은 각자 개인 — **RLS** 가 `auth.uid()` 기준으로 데이터를 분리해 서로 안 보인다.
> - 배포(GitHub Pages)도 같은 프로젝트를 쓴다(키는 repo Secrets, §6).

> [!tip] 역할별 진행
> - **팀원(개발)** → §1 → §2(키 2개) → §3 실행. **§4 는 건너뜀**.
> - **소유자/배포 담당** → §4(백엔드 1회 셋업) + §5, §6 + repo Secrets 등록

```bash
git clone <repo-url> lumiverse
cd lumiverse
npm install
```

## 2. 환경변수 (`.env.local`)

```bash
cp .env.example .env.local
```

`.env.local` 에 **소유자가 공유한** 키 2개를 넣는다(팀 전원 동일 값):

```
VITE_SUPABASE_URL=https://<공유-project-ref>.supabase.co
VITE_SUPABASE_ANON_KEY=<공유 anon public key>
```

- 값은 **프로젝트 소유자에게 요청**. anon 키는 공개 전제(RLS 로 보호)라 팀 내 공유 OK.
- ⚠️ `.env.local` 은 **커밋 금지**(.gitignore 처리됨). `service_role`·LLM 키는 **절대 여기 넣지 말 것** — 그건 Edge Function secrets 전용(소유자만, §4).

> [!note] 값이 비면 흰 화면
> `VITE_SUPABASE_URL`/`ANON_KEY` 가 비어 있으면 supabase-js 가 로드 시 예외를 던져 앱이 흰 화면이 된다. 반드시 채울 것.

## 3. 로컬 실행

```bash
npm run dev        # http://localhost:5173
```

- 가입/로그인(이메일) → 모드 → 목표 → AI 분해 → 내 우주(별 체크) 흐름 확인.
- 로그인이 안 되면 → §7 "이메일 확인" 참고.

## 4. 백엔드 셋업 (Supabase) — 소유자만, 1회

> 팀원은 이 절을 **건너뛴다**. 공유 프로젝트에 이미 셋업돼 있으니 §2 의 키 2개만 받으면 끝.

소유자가 **공유 프로젝트 하나**에 1회만 셋업한다. 리포의 `supabase/migrations` 가 스키마를 재현한다. LLM 호출만 Edge Function, 데이터 CRUD는 `supabase-js` 직접(RLS). 상세 [supabase/README.md](../supabase/README.md).

```bash
# 1) 로그인 & 프로젝트 연결 (대화형 — 직접 실행)
npx supabase login
npx supabase link --project-ref <공유-project-ref>

# 2) 스키마 적용 (DB 비밀번호 물어봄)
npx supabase db push
#   → Table Editor 에 lumiverse_* 테이블 6종 확인

# 3) LLM 키 등록 (Edge Function secrets · 클라이언트 노출 금지)
#    Solar 키: https://console.upstage.ai/api-keys (소유자 키 1개로 운영)
npx supabase secrets set LLM_API_KEY=<solar key> \
  LLM_BASE_URL=https://api.upstage.ai/v1 LLM_MODEL=solar-pro2

# 4) 함수 배포
npx supabase functions deploy goal-breakdown
```

> [!note] 인증 설정 (Dashboard, 1회)
> - **Authentication → Providers → Email**: 데모 편의를 위해 **Confirm email OFF** (켜면 가입 후 메일 확인 필요 → 현재 앱 흐름이 막힘).
> - **카카오**(선택): Providers → Kakao 활성화 + 카카오 앱 REST 키/시크릿 + Redirect URI `https://<ref>.supabase.co/auth/v1/callback`.
> - **URL Configuration**: `http://localhost:5173`, `http://127.0.0.1:5173`, `https://inorrni.github.io/lumiverse/` 등록.

> [!tip] LLM 공급자 전환 (7월 Groq)
> 코드 변경 없이 secrets 3개만 교체:
> `LLM_BASE_URL=https://api.groq.com/openai/v1 LLM_MODEL=llama-3.3-70b-versatile LLM_API_KEY=<groq key>`

> [!tip] LLM 공급자 전환 (7월 Groq)
> 코드 변경 없이 secrets 3개만 교체:
> `LLM_BASE_URL=https://api.groq.com/openai/v1 LLM_MODEL=llama-3.3-70b-versatile LLM_API_KEY=<groq key>`

## 5. 빌드

```bash
npm run build      # dist/ 생성
npm run preview     # 빌드 결과 로컬 확인
```

- 빌드 시 Vite `base`가 자동으로 `/lumiverse/` (배포 서브패스). 로컬 dev는 `/`.
- 통과 기준: 에러 0. 번들에 `service_role`·LLM 키가 없어야 함(절대 import 금지).

## 6. 배포 (GitHub Pages)

**`main` 에 push/머지하면 자동 배포** — `.github/workflows` 의 Actions가 `npm ci → build → Pages 배포` 수행(SPA용 `404.html` 폴백 생성).

```bash
# 작업 → PR → main 머지  (직접 main 푸시 금지, 리뷰 후 머지)
```

- 진행 상황: GitHub 저장소 → **Actions** 탭.
- 라이브: **https://inorrni.github.io/lumiverse/**
- 수동 트리거: Actions → Deploy to GitHub Pages → **Run workflow**.

> [!important] 배포도 같은 공유 프로젝트를 쓴다
> Vite 는 빌드 시 `VITE_*` 를 번들에 박는다. 라이브가 붙을 백엔드는 **repo Secrets** 에 등록된 키 = **개발에 쓰는 그 공유 프로젝트와 동일**(소유자 프로젝트).
> - 등록 위치: Settings → **Secrets and variables → Actions** → `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (소유자가 1회).
> - 워크플로우([deploy.yml](../.github/workflows/deploy.yml))의 build step `env:` 가 이 Secrets 를 주입한다 — **등록만 하고 주입 안 하면 배포본이 백엔드에 연결 안 됨**.
> - 즉 로컬·배포 모두 **한 프로젝트**를 바라본다. 옮기려면 그 프로젝트로 §4 셋업 후 repo Secrets 의 두 값만 교체.

## 7. 자주 막히는 곳

| 증상 | 원인 / 해결 |
|---|---|
| 로그인이 안 됨 ("Email not confirmed") | Confirm email 이 ON. Dashboard에서 **OFF** (§4 인증 설정) |
| 분해가 "AI가 …하지 못했어요" | LLM secrets 미등록/오타, 또는 모델이 `solar-pro2` 아님(JSON 모드는 solar-pro2 전용) |
| 데이터가 안 보임/저장 안 됨 | `.env.local` URL·anon 키 확인, 로그인 상태 확인(RLS가 비로그인 차단) |
| 한동안 쓰다 갑자기 죽음 | Supabase 무료티어 **7일 미접속 일시정지** — 대시보드에서 깨우기 |
| 배포 페이지가 빈 화면/404 | 서브패스(`/lumiverse/`) 문제 — `base`·라우터 `basename` 확인, 새로고침은 `404.html` 폴백 |
| 카카오 로그인 실패 | Dashboard provider·Redirect URI·URL Configuration 미설정 |
