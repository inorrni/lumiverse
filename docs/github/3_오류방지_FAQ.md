# 3. 오류방지 / 트러블슈팅 FAQ

> [!tip] 만능 해결책
> 상황을 그대로 설명하고 **에러 메시지를 통째로 복사해** Claude에게 주세요.
> 예: "이런 에러가 났어, 무슨 뜻이고 어떻게 고쳐? (에러 붙여넣기)"
> 위험한 작업이 필요하면 Claude가 먼저 설명하게 돼 있습니다([1번 문서 안전수칙](1_깃_협업_클로드코드.md)).

---

## 상황별 빠른 대처

### 😵 main에서 그냥 작업해버렸어요 (브랜치를 안 팠음)
- **증상**: 새 브랜치 안 만들고 `main`에서 코드를 고침.
- **원인**: 시작할 때 브랜치 생성을 건너뜀.
- **Claude에게**:
  ```
  실수로 main에서 작업했어. 아직 커밋 안 한(또는 커밋만 하고 푸시 전인) 변경을
  새 브랜치 feat/xxx 로 옮기고, main은 깨끗한 상태로 되돌려줘.
  ```

### ↩️ 방금 커밋을 되돌리고 싶어요
- **증상**: 잘못 커밋함.
- **원인**: 너무 일찍/잘못 묶어 커밋.
- **Claude에게**:
  ```
  방금 한 커밋을 되돌리고 싶어. 변경 내용은 그대로 두고 커밋만 취소해줘. (아직 푸시 전이야)
  ```
  > 푸시했는지 여부를 꼭 알려주세요 — 처리 방법이 달라집니다.

### 💥 pull 했더니 충돌(conflict)이 났어요
- **증상**: "CONFLICT", "Merge conflict in ..." 메시지. 파일에 `<<<<<<<`, `=======`, `>>>>>>>` 표시.
- **원인**: 같은 부분을 나와 남이 다르게 고침.
- **Claude에게**:
  ```
  pull 하다가 충돌났어. 어떤 파일이 충돌인지 보여주고,
  각 부분이 무슨 뜻인지 설명하면서 어떤 걸 남길지 같이 정리해줘.
  ```
  > 혼자 표시(`<<<<`)를 손으로 지우다 더 꼬이기 쉽습니다. Claude와 함께 푸세요.

### 🚫 push가 거부됐어요 (rejected / non-fast-forward)
- **증상**: `! [rejected] ... (fetch first)` 또는 `Updates were rejected`.
- **원인**: 내가 푸시하기 전에 원격에 다른 변경이 들어옴.
- **Claude에게**:
  ```
  푸시가 rejected 됐어. 원격 최신 변경을 안전하게 가져와 합친 다음 다시 푸시해줘.
  ```
  > ⚠️ "force로 밀어버려"는 **하지 마세요.** 남의 작업을 지울 수 있습니다.

### 🧭 "detached HEAD" 상태래요
- **증상**: 브랜치가 아니라 특정 커밋에 떨어져 있다는 안내.
- **원인**: 과거 커밋을 직접 체크아웃함.
- **Claude에게**:
  ```
  detached HEAD 상태가 됐어. 내 작업 안 잃게 안전하게 브랜치로 돌아가게 해줘.
  ```

### 🗑️ 파일을 실수로 지웠거나 변경을 날렸어요
- **증상**: 방금까지 있던 코드가 사라짐.
- **Claude에게**:
  ```
  OO 파일(또는 변경)을 실수로 날린 것 같아. 마지막 커밋 시점으로 복구할 수 있어?
  ```
  > **커밋해 둔 것**은 거의 항상 복구됩니다. 그래서 자주 커밋하는 게 안전장치입니다.

### 🔑 키나 .env를 커밋한 것 같아요
- → [2_보안_체크리스트.md 의 "실수로 키를 커밋했다면"](2_보안_체크리스트.md) 절차를 따르세요. **키 폐기·재발급이 1순위.**

## 사고를 미리 막는 습관

- **자주 커밋한다** — 되돌릴 지점이 많아집니다.
- **작게 PR 한다** — 충돌이 작고 리뷰가 쉽습니다.
- **시작 전 main 최신화** — 충돌의 가장 큰 원인을 줄입니다.
- **위험 명령은 멈추고 묻는다** — `force`, `reset --hard`, 브랜치 삭제는 항상 한 번 더 확인.
- **혼자 끙끙대지 않는다** — 막히면 바로 Claude나 팀원에게.
