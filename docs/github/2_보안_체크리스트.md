# 2. 보안 체크리스트

가장 흔하고 가장 아픈 사고는 **비밀키를 실수로 GitHub에 올리는 것**입니다. 한 번 올라간 키는 "지웠어도" 기록(history)에 남으므로, 곧바로 폐기·재발급해야 합니다. 아래만 지키면 대부분 예방됩니다.

---

## 절대 커밋하면 안 되는 것

- `.env`, `.env.local` 등 **환경변수 파일**
- **LLM API 키(Groq·Solar 등)**, Supabase 키/토큰, 그 외 모든 비밀번호·토큰
- 개인 인증서(`*.pem` 등)

> `.gitignore`가 이 파일들을 막아주지만, **방심 금지**. 커밋·푸시 전에 항상 한 번 확인하세요.

---

## 이 프로젝트의 키 구조 (서버측 보관)

Lumiverse는 **LLM 키를 백엔드(FastAPI) 서버 환경변수로만** 보관합니다:

- **LLM 키(Groq·Solar)** 는 별도의 백엔드 리포에서 **서버 환경변수**로만 다룹니다. 이 프론트엔드 리포·빌드·브라우저 어디에도 들어가지 않습니다.
- 이 프론트엔드 `.env.local`에는 **Supabase URL·anon key**(공개용 — RLS로 보호되므로 노출돼도 안전)와 **백엔드 API 주소**만 둡니다.
- 즉 **비밀키는 클라이언트로 절대 내려보내지 않습니다.**

> [!warning] Vite 함정 — `VITE_` 변수는 노출된다
> Vite에서 `VITE_`로 시작하는 환경변수는 **빌드 시 클라이언트(브라우저) 코드에 그대로 박혀** 누구나 볼 수 있습니다.
> 따라서 **LLM 키 같은 비밀값을 `VITE_`로 넣어 배포하면 안 됩니다.** 그건 공개와 같습니다.
> `VITE_`에는 Supabase anon key처럼 "공개돼도 안전한 값"만 둡니다. 진짜 비밀키는 백엔드 서버 환경변수에만 둡니다.

---

## 푸시 전 점검 (매번)

Claude에게 이렇게 확인할 수 있습니다:

```
푸시하기 전에, 이번 변경분에 API 키나 비밀값, .env 파일이 섞여 있지 않은지 확인해줘.
```

직접 확인하려면:

```bash
git status              # .env.local 같은 파일이 목록에 뜨면 안 됨
git diff --staged       # 커밋될 내용에 키 문자열이 없는지 훑기
```

## 🚨 실수로 키를 커밋/푸시했다면

순서가 중요합니다. **"기록 지우기"보다 "키 폐기"가 먼저**입니다.

1. **즉시 해당 키를 폐기·재발급**한다.
   - Groq/Solar/Supabase: 각 대시보드에서 그 키를 **삭제(revoke)** 하고 새 키 발급.
   - 이미 외부에 노출됐다고 가정하세요. 재발급이 가장 확실한 방어입니다.
2. **코드/기록에서 제거**한다 — Claude에게:
   ```
   실수로 OO 파일에 키를 커밋했어. 키는 이미 폐기·재발급했고,
   이 파일을 추적에서 빼고 .gitignore에 넣은 뒤, git 기록에서도 제거하는 방법을 알려줘.
   ```
   - 단순히 다음 커밋에서 지워도 **과거 기록엔 남습니다.** 기록까지 지우려면 history 정리가 필요하고, 이는 force push를 동반하므로 **반드시 팀에 먼저 공지**하고 함께 진행하세요.
3. **팀에 알린다** — 누가 그 키를 쓰고 있었는지, 재발급으로 영향받는 곳이 없는지 확인.
