## 1. 협업 흐름 한눈에

우리는 **`main` 직접 수정 금지 → 각자 브랜치에서 작업 → PR로 합치기** 방식을 씁니다.

```
  ┌─ main (항상 동작하는 최신본, 보호됨) ─────────────────────────┐
  │                                                               │
  │   ① main 최신화        ④ 푸시          ⑤ PR 생성   ⑥ 머지 │
  │         │                  │                │          │      │
  └─────────┼──────────────────┼────────────────┼──────────┘      │
           ▼                  ▲                ▼                 ▲
        ② 브랜치 생성 ──→ ③ 작업·커밋 ──→ feat/내-작업 ──→ 리뷰 후 main 으로 합침
```

| 단계 | 무엇을 | Claude에게 이렇게 |
|---|---|---|
| ① 최신화 | 시작 전 main을 최신으로 | "main 최신화해줘" |
| ② 브랜치 | 내 작업 공간 만들기 | "feat/goal-input 브랜치 만들어줘" |
| ③ 작업·커밋 | 코드 짜고 저장 | "변경분 용도별로 커밋해줘" |
| ④ 푸시 | 내 브랜치를 GitHub에 올림 | "내 브랜치 푸시해줘" |
| ⑤ PR | 합쳐달라는 요청 | "현재 브랜치로 PR 올려줘" |
| ⑥ 머지 | 상대가 확인 후 main에 합침 | (GitHub에서 리뷰 후) |

> 자세한 프롬프트는 [docs/1_깃_협업_클로드코드.md](1_깃_협업_클로드코드.md).

## 2. 브랜치 규칙

- **`main` 은 신성불가침** — 직접 커밋/푸시하지 않습니다. 항상 동작하고 배포 가능한 상태로 유지합니다.
- **작업은 항상 새 브랜치에서** 하고, 끝나면 PR로 `main`에 합칩니다.
- **브랜치 이름**은 `종류/짧은-설명` 형식 (영어 소문자, 띄어쓰기 대신 `-`):

| 종류 | 언제 | 예시 |
|---|---|---|
| `feat/` | 새 기능 | `feat/goal-input` |
| `fix/` | 버그 수정 | `fix/star-color` |
| `docs/` | 문서만 | `docs/readme-update` |
| `chore/` | 설정·잡일 | `chore/eslint-config` |

- **작업 시작 전 반드시 `main`을 최신화**한 뒤 브랜치를 만듭니다. (오래된 main에서 시작하면 충돌이 늘어납니다.)

---

## 3. 커밋 규칙

CLAUDE.md의 "깃푸시" 규칙과 동일합니다. **이 규칙을 그대로 Claude에게 전달**하면 됩니다:

> 리포 루트에서 변경분을 용도별로 분류해서 커밋해줘.
> 한 커밋에 한 가지 목적만 — 타입(feat/fix/docs/...)이 다르면 커밋도 나눠.
> 메시지는 변경 내용 보고 Conventional Commits로 알아서 작성.

- **한 커밋 = 한 가지 목적.** 기능과 문서 수정을 한 커밋에 섞지 않습니다.
- 메시지는 **Conventional Commits** 형식: `타입: 설명`
  - `feat: 목표 입력 화면 추가`
  - `fix: 별 색상이 안 바뀌는 문제 수정`
  - `docs: 협업 가이드 추가`
  - 그 외 `chore`(설정), `refactor`(동작 그대로 정리), `style`(포맷).

## 4. PR(Pull Request) 규칙

- **작게, 자주.** 큰 PR 하나보다 작은 PR 여러 개가 리뷰도 쉽고 충돌도 적습니다.
- PR 본문은 [PR 템플릿](.github/pull_request_template.md)이 자동으로 떠요 — 그대로 채웁니다.
- **셀프 머지 지양** — 가능하면 **상대가 한 번 보고** 머지합니다. (배우는 단계에선 서로 봐주는 게 가장 좋은 안전장치입니다.)
- 머지 방식은 **Squash merge** 권장 — 브랜치의 커밋들을 하나로 합쳐 `main` 히스토리를 깔끔하게 유지합니다.
- 머지된 브랜치는 삭제합니다 (GitHub이 버튼으로 안내).

## 5. 충돌(conflict) 방지

충돌은 "같은 부분을 두 사람이 동시에 고쳤을 때" 생깁니다. 다음으로 대부분 예방됩니다:

- **작업을 나누기** — 가능하면 서로 다른 파일/영역을 맡습니다.
- **자주 합치기** — PR을 작게 쪼개 빨리 머지합니다. 브랜치를 며칠씩 묵히지 않습니다.
- **시작 전 최신화** — 새 브랜치는 항상 최신 `main`에서 시작합니다.

충돌이 났을 때 푸는 법은 → [docs/3_오류방지_FAQ.md](3_오류방지_FAQ.md). **혼자 무리하지 말고 Claude에게 "충돌났어, 도와줘"라고 하세요.**

## 6. 금지 사항 · 보안 요약

- 🚫 **`.env`·API 키·토큰을 커밋하지 않습니다.** (`.gitignore`가 막아주지만 항상 확인)
- 🚫 **`main`에 직접 푸시 / force push 하지 않습니다.**
- 🚫 잘 모르는 위험 명령(`reset --hard`, `push --force` 등)을 그냥 실행하지 않습니다 — **멈추고 물어보기.**

> 키를 실수로 올렸을 때 대처 등 상세는 → [docs/2_보안_체크리스트.md](2_보안_체크리스트.md)

## 7. 관리자용 — 레포 1회 세팅 (저장소 소유자만)

아래는 GitHub 웹에서 한 번만 설정합니다. (코드가 아니라 GitHub 설정입니다.)

1. **협업자 초대** -> 진행중
   `Settings → Collaborators → Add people` 로 팀원 GitHub 계정 초대 → 팀원이 메일/알림에서 수락.

2. **[`main` 브랜치 보호](https://billtech.tistory.com/31)** (사고 차단의 핵심) -> 전체 완료
   `Settings → Branches → Add branch ruleset (또는 Add rule)` 에서 `main` 대상으로:
   - ☑ Require a pull request before merging (직접 푸시 금지, PR 필수)
   - ☑ (선택) Require approvals: 1 — 머지 전 리뷰 1건 필요
   - ☑ Block force pushes
   → 이 설정으로 "실수로 main을 날리는" 사고가 구조적으로 막힙니다.

2. **Secret Scanning 켜기** -> 활성화
   `Settings → Code security → Secret protection` 에서 활성화 → 키가 실수로 올라가면 GitHub이 알려줍니다.
