# 셋업 · 빌드 · 배포 가이드

> [!abstract] 이 문서로 할 수 있는 것
> 새 팀원이 **클론 → 실행 → (백엔드) → 빌드 → 배포**까지 막힘 없이 따라오게 하는 한 장짜리 안내.
> 스택: **Vite 5 · React 18 · React Router 7 / Supabase(PostgreSQL·Auth·Edge Functions) · Solar(LLM)**.

---

## 0. 사전 준비

| 필요 | 버전/비고 |
|---|---|
| Node.js | **20 이상** (CI는 20) · `node -v` |
| npm | Node 동봉 |
| Git | GitHub 저장소 접근 권한 |
| Supabase 계정 | **전원**(개발 단계는 각자 본인 프로젝트를 쓴다 — §4) |

> [!important] 개발 / 배포 구도 (먼저 이해)
> - **개발**: 팀원 **각자 본인 Supabase 프로젝트**를 만들어 붙는다. 서로 데이터·스키마 영향 없이 자유롭게 실험. → §2, §4 를 **각자 1회** 셋업.
> - **배포(GitHub Pages)**: **공유 "prod" 프로젝트 1개**가 라이브를 담당. 이 키는 repo Secrets 에만 등록돼 Actions 빌드에 주입된다(팀원이 손댈 필요 없음).
>
> 즉 "내 로컬 = 내 프로젝트 / 배포 사이트 = 공유 prod" 로 분리한다.

> [!tip] 역할별 진행
> - **누구나(개발)** → §1 → §2 → §4(본인 프로젝트 셋업) → §3 실행
> - **배포 담당** → 위 + §5, §6 + 공유 prod 프로젝트의 키를 repo Secrets 에 등록(§6)

## 1. 클론 & 설치

```bash
git clone <repo-url> lumiverse
cd lumiverse
npm install
```

> [!warning] Claude Code 훅 신뢰
> 이 저장소엔 시크릿 스캔 등 훅이 있어, 클론 후 Claude Code 첫 실행 시 **훅 신뢰 승인**을 물어볼 수 있다. 내용 확인 후 승인.

## 2. 환경변수 (`.env.local`)

```bash
cp .env.example .env.local
```

`.env.local` 에 **본인 Supabase 프로젝트**의 키 2개를 넣는다(개발 단계 = 각자 프로젝트):

```
VITE_SUPABASE_URL=https://<본인-project-ref>.supabase.co
VITE_SUPABASE_ANON_KEY=<본인 anon public key>
```

- 프로젝트가 아직 없으면 → **§4-0** 에서 먼저 만든다.
- 값 위치: 본인 Supabase Dashboard → **Project Settings → API** (`Project URL`, `anon` `public`).
- ⚠️ `.env.local` 은 **커밋 금지**(.gitignore 처리됨). `service_role`·LLM 키는 **절대 여기 넣지 말 것** — 그건 Edge Function secrets 전용.

> [!note] 값이 비면 흰 화면
> `VITE_SUPABASE_URL`/`ANON_KEY` 가 비어 있으면 supabase-js 가 로드 시 예외를 던져 앱이 흰 화면이 된다. 반드시 채울 것.

## 3. 로컬 실행

```bash
npm run dev        # http://localhost:5173
```

- 가입/로그인(이메일) → 모드 → 목표 → AI 분해 → 내 우주(별 체크) 흐름 확인.
- 로그인이 안 되면 → §7 "이메일 확인" 참고.

## 4. 백엔드 셋업 (Supabase) — 전원, 각자 본인 프로젝트로 1회

개발 단계는 **각자 자기 프로젝트**에 붙는다(데이터·스키마 격리). 리포의 `supabase/migrations` 가 스키마를 그대로 재현해 주므로 아래만 따라오면 된다. LLM 호출만 Edge Function, 데이터 CRUD는 `supabase-js` 직접(RLS). 상세 [supabase/README.md](../supabase/README.md).

```bash
# 0) 본인 Supabase 프로젝트 생성 (Dashboard → New project)
#    Region: Northeast Asia (Seoul) 권장. DB 비밀번호는 따로 메모.
#    생성 후 Settings → API 의 URL·anon key 를 §2 .env.local 에 넣는다.

# 1) 로그인 & 프로젝트 연결 (대화형 — 직접 실행)
npx supabase login
npx supabase link --project-ref <본인-project-ref>

# 2) 스키마 적용 (DB 비밀번호 물어봄)
npx supabase db push
#   → Table Editor 에 lumiverse_* 테이블 6종 확인

# 3) LLM 키 등록 (Edge Function secrets · 클라이언트 노출 금지)
npx supabase secrets set LLM_API_KEY=<solar key> \
  LLM_BASE_URL=https://api.upstage.ai/v1 LLM_MODEL=solar-pro2

# 4) 함수 배포
npx supabase functions deploy goal-breakdown
```

> [!note] 인증 설정 (본인 Dashboard, 1회)
> - **Authentication → Providers → Email**: 데모 편의를 위해 **Confirm email OFF** (켜면 가입 후 메일 확인 필요 → 현재 앱 흐름이 막힘).
> - **카카오**(선택): Providers → Kakao 활성화 + 카카오 앱 REST 키/시크릿 + Redirect URI `https://<본인-ref>.supabase.co/auth/v1/callback`.
> - **URL Configuration**: `http://localhost:5173`, `http://127.0.0.1:5173` 등록.

> [!warning] Solar 키 주의
> Solar 키는 프로젝트가 아니라 **Upstage 계정**에 묶인다(종량제). 각자 본인 프로젝트를 써도 키는 **① 팀이 한 개를 공유**(사용량 한 계정 합산)하거나 **② 각자 발급**([console.upstage.ai/api-keys](https://console.upstage.ai/api-keys)) 중 택일.

> [!tip] LLM 공급자 전환 (7월 Groq)
> 코드 변경 없이 secrets 3개만 교체:
> `LLM_BASE_URL=https://api.groq.com/openai/v1 LLM_MODEL=llama-3.3-70b-versatile LLM_API_KEY=<groq key>`

## 5. 빌드

```bash
npm run build      # dist/ 생성
npm run preview     # 빌드 결과 로컬 확인
```

- 빌드 시 Vite `base`가 자동으로 `/lumiverse/` (배포 서브패스). 로컬 dev는 `/`.
- 통과 기준: 에러 0. 번들에 `service_role`·LLM 키가 없어야 함(절대 import 금지).

## 6. 배포 (GitHub Pages)

**`main` 에 push/머지하면 자동 배포** — `.github/workflows` 의 Actions가 `npm ci → build → Pages 배포` 수행(SPA용 `404.html` 폴백 생성).

```bash
# 작업 → PR → main 머지  (직접 main 푸시 금지, 리뷰 후 머지)
```

- 진행 상황: GitHub 저장소 → **Actions** 탭.
- 라이브: **https://inorrni.github.io/lumiverse/**
- 수동 트리거: Actions → Deploy to GitHub Pages → **Run workflow**.

> [!important] 배포는 공유 "prod" 프로젝트를 쓴다 (개발용 본인 프로젝트 아님)
> Vite 는 빌드 시 `VITE_*` 를 번들에 박는다. 라이브가 붙을 백엔드는 **repo Secrets** 에 등록된 prod 프로젝트다.
> - 등록 위치: Settings → **Secrets and variables → Actions** → `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY` (배포 담당이 1회).
> - 워크플로우([deploy.yml](../.github/workflows/deploy.yml))의 build step `env:` 가 이 Secrets 를 주입한다 — **등록만 하고 주입 안 하면 배포본이 백엔드에 연결 안 됨**.
> - prod 프로젝트에도 §4-2~4(db push·secrets·functions deploy)·인증 설정이 1회 돼 있어야 한다.

## 7. 자주 막히는 곳

| 증상 | 원인 / 해결 |
|---|---|
| 로그인이 안 됨 ("Email not confirmed") | Confirm email 이 ON. Dashboard에서 **OFF** (§4 인증 설정) |
| 분해가 "AI가 …하지 못했어요" | LLM secrets 미등록/오타, 또는 모델이 `solar-pro2` 아님(JSON 모드는 solar-pro2 전용) |
| 데이터가 안 보임/저장 안 됨 | `.env.local` URL·anon 키 확인, 로그인 상태 확인(RLS가 비로그인 차단) |
| 한동안 쓰다 갑자기 죽음 | Supabase 무료티어 **7일 미접속 일시정지** — 대시보드에서 깨우기 |
| 배포 페이지가 빈 화면/404 | 서브패스(`/lumiverse/`) 문제 — `base`·라우터 `basename` 확인, 새로고침은 `404.html` 폴백 |
| 카카오 로그인 실패 | Dashboard provider·Redirect URI·URL Configuration 미설정 |
