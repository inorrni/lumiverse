# 회의록 정리 (lumiverse)

카톡 대화 export(+첨부 사진)를 **날짜별 회의록 .md**로 정리한다. 산출물은 Obsidian에서 바로 읽히는 구조화 문서.

## 입력 위치
- **대화**: `yumi_docs/2.회의록/초안/`의 **오늘 날짜** 파일
  - KakaoTalk export (`.md` 또는 `KakaoTalk_Chat_*_YYYY-MM-DD-*.csv`). 텍스트는 `Date User Message`. CSV는 `Date,User,Message` 컬럼.
- **사진**: `yumi_docs/2.회의록/초안/attachments/`의 **오늘 날짜** 이미지 (`KakaoTalk_Photo_YYYY-MM-DD-*.{png,jpeg}`)
- "오늘 날짜"는 환경의 현재 날짜를 쓴다. 여러 날짜가 한 파일에 섞여 있으면 날짜별로 각각의 회의록을 만든다.

## 산출물
- 신규/갱신: `yumi_docs/2.회의록/초안/회의록초안_YYMMDD.md` (날짜별 1개. 기존 네이밍 규칙 준수: 6자리 YYMMDD)
- frontmatter: `tags:[회의록, 프로젝트/lumiverse]`, `날짜`, `참석자`, `작성근거`

## 정리 과정 (핵심 규칙)
1. **전부 포함, 잡담은 압축** — 대화·링크·사진을 빠짐없이 반영하되, 결정·근거·다음 할 일 중심으로. 농담/리액션은 흐름에 필요한 만큼만.
2. **링크는 인라인 `[]()`** — 대화 중 URL은 맥락 문장 안에 `[설명](url)` 형태로 삽입(맨 끝에 나열하지 말 것).
3. **스타일·맥락 파악 후 구조화** — 먼저 그날의 주제/결정을 파악하고 `> [!info]/[!important]` callout · 표 · 헤더로 재배열. (시간순 나열 X, 의미 단위로 묶기)
4. **사진 — "사진" 메시지 = 첨부 이미지, 순서로 매칭**:
   - 대화에서 메시지가 **`사진` 두 글자만** 있는 줄이 곧 **첨부된 이미지 1장**이다. (URL·`파일:` 첨부는 별개로, 사진 수에 포함하지 않음)
   - 사진은 **한 번에 일괄 다운로드**되므로 기본 전제는 **「대화의 `사진` 등장 순서 = attachments 파일 순서」**. → "사진" 메시지를 시간순으로 나열하고, attachments 파일을 **순서대로 1:1 매칭**해 각 사진에 그 메시지의 **`[HH:MM]` 캡션** + 한 줄 설명을 단다. Obsidian 임베드 `![[파일명]]`. 사진이 세로일 경우 `![[파일명|400]]`
   - ⚠️ **매칭 전 반드시 순서 일치를 검증**(절차 3-1 참고): 일부 export는 파일 순서가 어긋난다. 어긋나면 **이미지 내용으로 매칭**(폴백).
   - 파일명의 `YYYY-MM-DD-HH-MM-SS`는 개별 촬영시각이 아니라 **카톡 내보내기 시각(전부 동일)** 이니 시각 출처로 쓰지 말 것. 시각은 항상 "사진" 메시지 시간에서 가져온다.
   - 분 단위 매칭이 애매한 연속 캡처는 묶음/근사 시각(예:`[11:48–12:05]`)으로 정직하게 표기.
5. **변경 전-후 비교는 접이식 callout으로 아카이브** — 회의 중 뒤집힌 결정(예: 명칭/규칙 변경)은 `> [!note]- 변경 이력` 접이식 callout에 전↔후를 남긴다.
6. **빈 항목은 `[!check]` 제안으로 채움** — 정해지지 않은/누락된 항목은 `> [!check] 제안`으로 합리적 다음 단계를 제시(확정인 척하지 말 것).
7. **최상단 간략 보고 요약** — 문서 맨 위에 한 줄 요약(보고용) → 그 아래 `> [!summary]- 상세 요약` 접이식 callout에 핵심 결정 묶음을 담는다.
8. **사진 파일명을 내용 기반으로 변경** — 회의록 작성·검증이 끝나면 사진을 `순번_YYMMDD_내용.확장자` 형식으로 rename하고 노트 임베드도 함께 갱신한다.
   - 순번 = 사진(대화) 순서(2자리, `01`~), 내용 = 짧은 한글 식별어(예: `12_260609_손그림와이어프레임`).
   - 확장자는 원본 그대로 유지. rename 후 **노트 임베드(`![[...]]`)를 새 파일명으로 교체**하고, 임베드↔실제파일 1:1 정합을 반드시 확인.
   - ⚠️ **회전/오프바이원 주의**: 일괄 rename 시 매핑 배열이 한 칸 밀리면 모든 파일이 회전한다. 배열은 **명시적 인덱스**(`N[1]=...`)로 만들고, 제자리 이동은 **충돌 없는 순서**(예: 내림차순)로 하며, rename 직후 앵커 2~3개의 **내용↔파일명 일치**를 눈으로 검증한다.

## 문서 골격
```
---
tags: [회의록, 프로젝트/lumiverse]
날짜: YYYY-MM-DD
참석자: ...
작성근거: KakaoTalk export (YYMMDD) + 첨부 사진
---

# 회의록 — YYYY-MM-DD (한 줄 주제)

> 한 줄 요약(보고용)

> [!summary]- 상세 요약
> - 핵심 결정 1 / 2 / 3 …

---
> [!check] 제안           (빈 항목 채움)

## 다음 할 일             (- [ ] 체크리스트)

---
## 1. 결정 사항           (표 또는 [!important])

---
## 2. 논의 내용           (인용/헤더/표/callout, 링크는 인라인)
#### 주제
표 논의 흐름 (시간순) 시각-변화 
> [!note]- 변경 이력      (전↔후 비교, 있을 때만)

---
## 3. 공유 이미지 (시각순)
##### ([HH:MM] 캡션 + ![[파일]])
```

## 절차
1. 오늘 날짜로 초안 대화 파일 + attachments 사진을 찾는다. (없으면 사용자에게 확인)
2. 대화 전문을 읽어 날짜·주제·결정·뒤집힌 결정·링크를 추출하고, **`사진` 메시지를 시간순으로 나열**한다.
3. **사진 매칭 (순서 우선 + 검증)**:
   - 3-1. **검증**: attachments 파일을 번호순으로 두고, 내용이 명확한 앵커 2~3개(예: 손그림·특정 화면 캡처)로 "N번째 사진 메시지 ↔ N번째 파일"이 맞는지 확인.
   - 3-2. **맞으면**: 사진 메시지 순서대로 파일을 1:1 매칭(빠르고 정확).
   - 3-3. **어긋나면**: 사진을 전부 열어 내용으로 메시지와 매칭(폴백). 특정 불가한 사진은 본문 기록으로 대체.
4. 위 규칙대로 `회의록초안_YYMMDD.md` 작성. 날짜가 여럿이면 각각 생성.
5. 매칭이 불확실한 사진/시각은 보고할 때 정직하게 표시하고, 원본 카톡에서 확인 가능한지 사용자에게 안내.
6. (마무리) 사진을 `순번_YYMMDD_내용.확장자`로 rename(정리 과정 8) + 노트 임베드 갱신 → 임베드↔파일 정합 검증.

## 주의
- 산출물의 사실관계는 **초안 대화와 사진만** 근거로 한다(다른 기획 문서를 끌어와 추측 금지).
- 원본 `초안/`(csv·attachments)은 보존한다. 회의록은 `2.회의록/초안` 바로 아래에 둔다.
