# 평가보고서 — 2026.06.16

> [!abstract] 종합
> 프론트엔드 4대 규칙을 **구조적으로 충족**했고, Must 유저플로우의 **온보딩 + 핵심 루프**가 동작 수준으로 구현됐다. 이후 **백엔드(Supabase: DB+Auth+LLM 프록시)까지 실연동**해 가입→AI 분해(Solar)→별 생성→캐시까지 E2E 검증을 마쳤다(§7). 남은 관문은 **자동화 테스트·사람 손 QA·카카오 OAuth 실왕복·Should 기능**. **MVP 토대로서 합격**.

## 3. MVP 기능·플로우 평가

### 3-1. Must 유저플로우 커버리지

| 단계 | 상태 | 비고 |
|---|---|---|
| 랜딩 | ✅ 동작 | 로그인 여부·우주 보유 분기 |
| 로그인/가입 | ✅ 동작(실연동) | Supabase Auth — 이메일(가입 폴백)+카카오. localStorage→Supabase 세션 |
| 모드 선택 | ✅ 동작 | 강도(살살/스파르타) + 입력방식(추천/알아서) |
| 목표 입력 + D-Day | ✅ 동작 | 별=일수 계산, 14개↑ 별자리 안내 |
| AI 세부목표 분해 | ✅ 동작(실연동) | Solar(Edge Function) 실호출 + 캐시. 4상태 + 행성 수정/추가/삭제, 3~5개 |
| **★ 핵심 루프(매일 반복)** | ✅ 동작 | 오늘 투두·행성 상세 데일리 체크 → **선명도 실시간 상승** |
| 내 우주(대시보드) | ✅ 동작 | 행성 클러스터·Daily Stars·TODAY·Mid-Check 티저 |
| 설정 | ✅ 동작 | 알림 토글·데이터 초기화·로그아웃 (Could) |
| AI 중간점검 · 블랙홀 · 별자리 | ⏳ 자리표시 | Should — `준비 중` |

**Must 핵심 루프 100% 구현, Should는 의도적 보류.**

### 3-2. 2-2 핵심 기능 동작
"되게 만드는 것" 기준 충족: 온보딩~핵심 루프가 끊김 없이 이어지고, 체크가 선명도·별 개수·대시보드에 반영된다. 상태는 **Supabase(PostgreSQL)** 영속 — 별 체크는 낙관적 업데이트(즉시 반영)+DB write.

### 3-3. 2-3 테스트·엣지케이스
- 빈 상태: 우주 0개 / 오늘 별 0개 / 분해 결과 0개 → 전용 안내+CTA ✅
- 에러: 분해 실패 → 친화 메시지 + 재시도 ✅ (`실패` 키워드로 재현)
- 입력 검증: 이메일 형식·빈 목표·과거 디데이·행성 3~5개 ✅
- 체크 롤백: 같은 날 재탭 시 해제 ✅

## 7. 백엔드 연동 상태 (Supabase) — 실연동 완료

mock 데이터 계층을 **Supabase 실백엔드**로 교체. 범위 = DB + Auth + LLM 프록시(goal-breakdown). 데이터 모델은 기획서 v4의 **4단계 풀스키마**(galaxies/planets/stars), 모든 객체 `lumiverse_` 접두사.

| 항목 | 상태 | 비고 |
|---|---|---|
| 스키마·RLS | ✅ 적용 | 6테이블, 전 테이블 RLS(본인만), 가입 트리거, 인덱스, 일괄생성 RPC. `db push` Local=Remote |
| 인증 | ✅ 연동 | Supabase Auth — 이메일(가입 폴백) + 카카오(코드/리다이렉트 준비). 세션 복원·구독 |
| LLM 프록시 | ✅ 연동 | `goal-breakdown` Edge Function ACTIVE. **Solar 실호출** + `llm_cache` 캐시 + rate limit + 출력검증 |
| 데이터 계층 | ✅ 재구성 | GoalStore→4단계, 별 체크=낙관적 업데이트+DB write, 선명도 계산 분리 |
| 공급자 추상화 | ✅ | OpenAI 호환 래퍼 — Solar(`solar-pro2`, JSON모드)↔Groq env 교체만으로 전환 |

> [!success] E2E 검증 (실서버 스크립트)
> 가입→세션 → goal-breakdown(Solar, ~5.3s, **행성 4개 한국어**) → 2회차 **캐시 히트** → RPC **별 15개**(3행성×5일) 생성 → **profile 트리거** 동작. 전 구간 통과.

> [!note] 보안
> service_role·LLM 키는 **Edge Function secrets** 전용(클라이언트/번들 금지). anon 키만 노출, **RLS로 보호**.

**기존 리스크 해소**: "AI·인증·DB 전부 mock", "Supabase ↔ 2단계 구조 정합성" 우려는 본 연동으로 해소(4단계 채택 + deriveGoal 변환층).

## 8. 인증 흐름 안정화 (FK 위반 23503 → 가드·OAuth 정리)

실제 클릭 테스트에서 **은하 생성 RPC가 FK 위반(23503)으로 실패**하는 버그를 발견·해결. 표면은 데이터 문제(프로필 누락)였으나, 근본 원인은 **인증 가드 부재**로 미로그인/불완전 세션 상태에서도 보호되어야 할 RPC가 호출되던 **구조적 허점**이었다.

| 항목 | 상태 | 비고 |
|---|---|---|
| 데이터 백필 | ✅ | profiles 누락 행 백필 + 가입 트리거 멱등화(`on conflict do nothing`) 마이그레이션 |
| 인증 가드 | ✅ | `RequireAuth` — `ready` 확인 후 미로그인 시 `/login`. 보호 라우트 레이아웃화(`/`·`/login`만 공개) |
| OAuth 복귀 흐름 | ✅ | 카카오 `redirectTo`→`/login?postLogin=1`. 랜딩 깜빡임 제거, 로그인 화면 로딩 경유 |
| 뒤로가기 정합 | ✅ | `/mode` BackRow `/login`→`/` |

> [!success] 효과
> 미로그인 상태로는 `/breakdown`에 도달해 RPC를 호출하는 것 자체가 불가능해져, **23503/23502 류 인증-흐름 불일치 오류를 원천 차단**. OAuth 복귀도 별도 페이지 깜빡임 없이 매끄럽게 이어짐.

> [!success] 검증 완료
> 가드 리다이렉트(미로그인 `/breakdown`→`/login`)·카카오 복귀 흐름(랜딩 깜빡임 제거) **브라우저 수동 QA 통과**. 백필 SQL은 앱이 붙는 프로젝트 ref에서 실행 전제.
