### 3-2. 디자인 토큰 (규칙 ①)
`LUM` 팔레트(흑백 먹 + 갱지)를 **CSS 변수 6종 카테고리**로 추출:
Color · Typography · Spacing(4px 그리드) · Radius · Shadow · Z-index.
모든 컴포넌트 CSS는 `var(--token)` 만 참조(raw hex/px 금지). 다크가 기본 테마.

### 3-3. 컴포넌트 분리 (규칙 ②)
`ui → layout → feature → pages` 계층. 페이지는 로직 없이 **조립만** 담당.
모든 인터랙티브 요소 터치 타깃 44px↑.

### 3-4. 4상태 UI (규칙 ③)
`DataView`(로딩·에러·빈·성공) + `useAsync` + `friendlyError`(401/429/500/네트워크 매핑).
AI 목표 분해 화면에서 4상태를 모두 구현. 로딩=스켈레톤, 에러=재시도, 빈=CTA.

### 3-5. 핵심 루프 (Must)
- `GoalStore.toggleStarToday`: 오늘의 별 체크 토글 → `done`/`lastCheck` 갱신 → **clarity 재계산**
- 대시보드 Daily Stars·행성 pip·행성 상세 선명도에 **즉시 반영**
- 같은 날 재탭 = 해제(롤백), 날짜 바뀌면 다시 체크 가능

### 3-6. 온보딩 플로우
랜딩(로그인 여부 분기) → 로그인(mock) → 모드(강도·입력방식) → 목표+D-Day → AI분해.

## 4. 핵심 의사결정

1. **목업 보존** — `screens/`까지 `src/sample/`로 옮겨 디자인 레퍼런스로 남기고 `/design` 경로에서 열람. 실제 앱은 새로 구성.
2. **데이터 모델 2단계** — 목업의 3단계(은하→행성→별-task)를 MVP에서 **목표(은하)→세부목표(별)** 2단계로 단순화. 체크 단위 = 세부목표. 행성 상세 = 목표 상세로 매핑.
3. **CSS Modules + 토큰** — 매우 커스텀한 우주 미감(그레인·먹·갱지) 때문에 Tailwind 대신 CSS 변수 토큰 + 컴포넌트 스코프 CSS 채택.
4. **mock 우선** — AI 분해·인증은 인터페이스를 고정한 mock으로 구현(2-1에서 Solar/Claude·Supabase로 교체 예정). localStorage 영속으로 새로고침에도 우주 유지.
5. **개발 주석 제외** — 목업의 `Annot`(API 엔드포인트·폴백 등 개발 메모)은 사용자 UI에서 제거, 의미 있는 동작만 구현.

## 7. 트러블슈팅 메모

- **JSX는 `//` 주석이 안 통한다** — `// <DesignCanvasApp />`가 텍스트+엘리먼트로 파싱돼 컴포넌트가 그대로 마운트됐다. 진짜 제거하려면 라인을 지워야 함.
- **KST 타임존 버그** — `toISOString()`은 UTC라 한국시간 자정에 날짜가 하루 밀린다. `lib/date`에서 로컬 달력 기준으로 통일.
- **dynamic+static 동시 import 경고** — DesignCanvasApp을 main에서 정적 import 하면 코드 분리가 풀린다. `/design` 지연 로드만 남기니 별도 청크(61.8kB)로 정상 분리.

## 9. 백엔드 구성 (Supabase) — mock → 실연동

> [!note] 한 줄 요약
> mock(localStorage·가짜 AI)을 **Supabase 실백엔드**로 교체했다. DB(PostgreSQL+RLS)·인증(이메일+카카오)·LLM 프록시(Edge Function, **Solar 실연동**)를 구성하고, 가입→AI 분해→별 생성→캐시→프로필 트리거까지 **엔드투엔드 검증 완료**.

### 9-1. 설계 결정 — 4단계 풀스키마 채택
- MVP 프론트는 2단계(목표→세부목표, done 카운터)로 단순화돼 있었으나, **백엔드는 기획서 아키텍처 v4의 4단계 풀스키마**로 확정: `User → Galaxy(목표) → Planet(세부목표) → Star(날짜별 투두)` + Constellation + llm_cache.
- 화면은 그대로 두기 위해 GoalStore 에 `deriveGoal` 변환층을 둬, DB의 galaxies/planets/stars 를 기존 `goals/steps` 형태로 변환해 소비. PostgREST 임베드 별칭(`planets:lumiverse_planets`)으로 코드 키 이름 유지.
- **네임스페이스**: 한 Supabase 프로젝트에 여러 앱이 공존할 수 있어 모든 객체에 **`lumiverse_` 접두사**(테이블 6종·함수 2종·트리거).

### 9-2. 구성 (범위 = DB + Auth + LLM 프록시)
| 영역 | 산출물 | 내용 |
|---|---|---|
| **스키마·RLS** | `supabase/migrations/…_init.sql` | 6테이블, 전 테이블 RLS(`auth.uid()` 본인만), 가입 트리거(profile 자동생성), 인덱스, 일괄생성 RPC |
| **인증** | `src/store/AuthStore.jsx` · `src/lib/supabase.js` | mock→Supabase Auth(세션 복원/구독), 이메일(가입 폴백)+카카오 OAuth |
| **LLM 프록시** | `supabase/functions/goal-breakdown` | OpenAI 호환 래퍼로 Solar↔Groq env 전환, JWT 인증·`llm_cache` 캐시·rate limit·출력검증(planets 3~5) |
| **데이터 계층** | `src/store/GoalStore.jsx` · `lib/{clarity,stars}.js` | 4단계 재구성, 별 체크=낙관적 업데이트+DB write, 선명도 계산 |
| **API** | `src/lib/api.js` | mock→`functions.invoke('goal-breakdown')` 실연동 |

### 9-3. LLM — Solar 연결 (공식문서 기준 정정)
- 엔드포인트 `https://api.upstage.ai/v1` (OpenAI 호환 `/chat/completions`), Bearer 인증.
- **`response_format: json_object` 는 `solar-pro2` 에서만 지원** → 기본 모델을 `solar-pro2` 로 확정(초안의 `solar-1-mini-chat` 은 구버전·JSON 모드 미지원이라 폐기).
- 키는 **Edge Function secrets**(`LLM_API_KEY`/`LLM_BASE_URL`/`LLM_MODEL`)에만 — 클라이언트/번들 금지. 7월 Groq 전환은 env 3개 교체만으로(코드 불변).

### 9-4. 검증 (E2E, 실서버)
- `supabase db push` → 6테이블 적용(Local=Remote), `functions deploy goal-breakdown` ACTIVE, secrets 등록 확인.
- 스크립트로 **가입→세션→goal-breakdown(Solar, ~5.3s, 행성 4개 한국어)→2회차 캐시 히트→RPC 별 15개(3행성×5일)→profile 트리거** 전 구간 통과.
- `npm run build` ✅ (164 모듈, 번들에 service_role 미포함).

## 10. 인증 가드 · OAuth 흐름 안정화 (FK 위반 23503 디버깅)

> [!note] 한 줄 요약
> 실제 클릭 테스트 중 **"우주로 떠나기"가 FK 위반(23503)으로 실패**. 원인을 추적해보니 단순 데이터 문제가 아니라 **인증 가드 부재로 화면 흐름과 세션 상태가 어긋나** 미로그인/스테일 세션 상태에서 RPC가 불리는 구조적 허점이었다. 가드·OAuth 복귀 흐름·뒤로가기를 정리해 재발을 차단했다.

### 10-1. 증상 · 진단
- `POST /rpc/lumiverse_create_galaxy_with_planets` → **409, code 23503** "Key is not present in table lumiverse_profiles" (galaxies→profiles FK 위반).
- **에러 코드가 단서**: 미로그인이면 `auth.uid()=null`→not-null(23502)이어야 함. 23503이 났다는 건 **세션은 유효한데 그 uid의 프로필 행만 없다**는 뜻.
- 근본 원인: `App.jsx`의 `/mode`·`/goal`·`/breakdown`·`/app/*`가 **전부 공개 라우트** → 세션 불완전 상태로도 온보딩을 끝까지 통과해 RPC 호출 가능. 카카오 복귀 후 `/root/#`에 멈추고, 뒤로가기가 `/login`으로 하드코딩돼 흐름이 꼬이는 게 방아쇠.

### 10-2. 조치
| 구분 | 내용 |
|---|---|
| **데이터 정합성** | `…_backfill_profiles.sql` — auth.users 중 profiles 누락 행 백필 + 가입 트리거 `on conflict do nothing` 멱등화 |
| **인증 가드** | `components/RequireAuth.jsx` 신규. `AuthStore.ready`(세션 복원 완료) 확인 후 미로그인이면 `/login` 리다이렉트. `App.jsx`에서 보호 라우트를 레이아웃 라우트로 묶음(`/`·`/login`만 공개) |
| **OAuth 복귀** | 카카오 `redirectTo`를 **`/login?postLogin=1`**로(랜딩 경유 X). `AuthPage`가 처리 중 폼 대신 **Spinner+"우주로 들어가는 중…"**만 노출하다 `ready&&user`면 다음 화면으로 |
| **뒤로가기** | `ModePage` BackRow `to="/login"` → `to="/"` (가드로 `/mode`가 로그인 필수가 됨) |

### 10-3. 트러블슈팅 메모
- **OAuth 세션은 서버 상태** — `localStorage.clear()`는 브라우저 토큰만 지움. 카카오는 같은 계정→**같은 uid**라 재로그인으로 누락 프로필이 새로 안 생긴다(트리거는 신규 insert 시만). → 백필이 정답.
- **공유 단일 프로젝트 주의** — 백필 SQL은 앱이 붙는 바로 그 프로젝트 ref에서 실행해야. 다른 개인 프로젝트에서 돌리면 `no rows`.
- **로딩이 너무 빨라 에러처럼 보임** — 처음엔 랜딩 경유+로딩 화면이었으나 한순간 깜빡여 오히려 불안해 보임 → **로그인 화면에 머물며 로딩만** 보여주는 방식으로 변경.
